## Project Overview

Embedded C firmware for a puzzle-solving pick-and-place machine (PREN2 project). Runs on an **STM32H753xx** (Cortex-M7) Nucleo-144 board. The MCU controls stepper motors (X/Y axes via DRV8886 drivers), pneumatic pistons (via IPS4260LM), and communicates over UART with a host system that sends puzzle piece coordinates.

## Build System

Uses CMake with Ninja and the `arm-none-eabi-gcc` cross-toolchain. The toolchain path is hardcoded for Windows in `cmake/gcc-arm-none-eabi.cmake` — adjust if building on macOS/Linux.

```bash
# Configure (Debug)
cmake --preset Debug

# Build
cmake --build build/Debug

# Configure + Build (Release)
cmake --preset Release && cmake --build build/Release
```

Output: `build/Debug/PuzzleSolver_MCU.elf`

There are no tests or lint commands — this is bare-metal firmware flashed to hardware.

## Architecture

### Entry Flow
`main.c` (STM32CubeMX-generated) → `Sys_Init()` → `App_Run()` (infinite loop)

- **`Core/Src/main.c`**: Auto-generated by STM32CubeMX. Only modify code inside `USER CODE BEGIN/END` blocks — CubeMX regeneration overwrites everything else. Peripheral init (GPIO, TIM2, TIM3, UART7) lives here.
- **`Core/Src/system/sys_init.c`**: Creates and wires all hardware objects (steppers, piston) with their GPIO pin assignments. This is where new peripherals get instantiated.
- **`Core/Src/system/app.c`**: Application logic entry point (`App_Run`). Also contains the TIM2 ISR callback that drives `StepGenerator_Update()`.

### Module Layers

**Actuators** (`Core/Src/actuators/`, `Core/Inc/actuators/`):
- `stepper.c/.h` — Low-level stepper motor driver (step/dir/enable/microstep GPIO control). Object-oriented C pattern using `Stepper_t` structs.
- `step_generator.c/.h` — Timer-driven step pulse generation with trapezoidal acceleration (DDA algorithm). Called from TIM2 ISR at fixed interval. Operates on `MoveBlock_t` structs that define motion profiles (accel/decel ramps, cruise speed).
- `piston.c/.h` — Pneumatic piston control with state machine (START→GRAB→MOVE→RELEASE). Timed transitions using TIM3.

**Coordination** (`Core/Src/coordination/`):
- `motion_planner.c/.h` — Converts mm-level coordinates to step counts (currently skeletal).

**Communication** (`Core/Src/communication/`):
- `package.c/.h` — Parses binary data packets from UART containing puzzle piece positions (start/end XY + rotation), up to 6 pieces per block.

**System** (`Core/Src/system/`, `Core/Inc/system/`):
- `sys_config.h` — Central `#define` configuration (feature toggles, timing constants, machine dimensions). All tunable parameters go here.
- `utils.h` — Shared `GPIO_Pin_t` type used across all modules.

### Key Patterns
- **Object-oriented C**: Modules use `*_Init(Self_t*, ...)` / `*_Method(Self_t*, ...)` with struct instances allocated in `sys_init.c`.
- **Conditional compilation**: Hardware features toggled via `CONFIG_*` defines in `sys_config.h` (e.g., `CONFIG_FOR_ENABLE_DRIVER`, `CONFIG_PISTON_HAS_LIMIT_SWITCH`).
- **ISR-driven motion**: TIM2 fires periodically → `StepGenerator_Update()` generates step pulses. TIM3 handles piston timing. Shared state uses `volatile`.

## Code Style

Enforced via `.clang-format` (Google-based, 2-space indent, 80-col limit). Include order: `main.h` first, then project headers (`"*.h"`), then standard library (`<...>`), then system headers.

## Important Files

- `PuzzleSolver_MCU.ioc` — STM32CubeMX project file (pin assignments, clock config, peripheral setup). Open with STM32CubeMX to modify hardware configuration.
- `STM32H753XX_FLASH.ld` — Linker script.
- `docs/` — Hardware datasheets (DRV8886 stepper driver, IPS4260LM piston driver, Nucleo board schematic, STM32H753 reference).
